## 摘要

本文档是校园搭子任务互助平台（Task Buddy）的软件设计规格说明书。系统采用微信小程序 + Node.js/Express + MySQL 的三层前后端分离架构，面向校园学生提供任务发布、搭子配对、进度监督、证据提交、互相评价与积分激励功能。本文档涵盖系统架构设计（模块划分、接口定义、技术选型）、界面设计（用户分析、界面规范、原型说明、交互流程）及详细设计（核心用例的完整流程、类设计、数据库 E-R 图与逻辑结构、关键算法），为后续编码实现提供直接依据。

## 2 系统架构设计

### 2.1 需求回顾

核心功能需求（P0）：微信授权登录注册（R01/R02）、任务贴发布与浏览（R03/R04）、搭子加入/退出（R07/R08）、证据上传（R10）、积分结算（R14/R15）、发布者放弃（R13）。

核心非功能需求：写操作响应 ≤ 1.5 秒、读操作 ≤ 1 秒、支持 200 并发、小程序主包 ≤ 2 MB、积分操作原子性、帖子内容 100% 接入微信内容安全接口。

### 2.2 架构选型分析

#### 2.2.1 备选架构模式对比

| 对比维度 | 方案 A：传统 MVC 单体 | 方案 B：前后端分离（本项目） |
|----------|----------------------|------------------------------|
| 前端渲染 | 服务端模板渲染 | 客户端渲染（微信小程序） |
| 接口形式 | 页面路由 + 表单提交 | RESTful JSON API |
| 部署方式 | 前后端同进程 | 前端在微信平台，后端独立部署 |
| 扩展性 | 前后端耦合，扩展困难 | 各层独立演进 |
| 适配微信小程序 | 不适合（小程序无浏览器渲染） | 天然契合 |
| 学习成本 | 低 | 中（需理解 REST 约定） |

**方案 C：Serverless（云函数）**：微信云开发可零运维，但免费配额有限，数据库能力弱，不适合复杂 SQL 查询（推荐算法、排名计算）。

#### 2.2.2 最终架构选择及理由

选择**方案 B：前后端分离架构**。理由：

1. 微信小程序本身就是独立客户端，强制前后分离；
2. RESTful API 便于分工开发和独立测试；
3. Node.js + Express 开发效率高，异步 I/O 适合高并发读操作；
4. MySQL 满足复杂查询（多表 JOIN、事务）和数据一致性要求。

**前端模块**：

| 模块 | 路径 | 职责 |
|------|------|------|
| 登录模块 | pages/login | 微信授权、首次绑定昵称头像 |
| 首页模块 | pages/home | 帖子列表展示、搜索筛选、推荐排序 |
| 帖子详情模块 | pages/post-detail | 任务全生命周期操作（加入/退出/证据/评价） |
| 发布模块 | pages/publish | 创建和编辑帖子 |
| 排行榜模块 | pages/ranking | 积分排名展示 |
| 个人中心模块 | pages/profile | 个人信息、历史帖子、积分记录 |
| API 服务层 | services/api.js | 封装所有 HTTP 请求，统一错误处理 |

**后端模块**：

| 模块 | 位置 | 职责 |
|------|------|------|
| 认证模块 | server.js: /api/auth/* | 微信 code 换 openid、用户注册登录 |
| 帖子管理模块 | server.js: /api/posts/* | 帖子 CRUD、状态流转 |
| 搭子互动模块 | server.js: /api/posts/:id/join,quit | 加入/退出搭子关系 |
| 证据评价模块 | server.js: /api/posts/:id/evidence,evaluate | 证据提交、互评处理 |
| 积分结算模块 | server.js: settlePost, recalcCompletionRate | 原子性积分操作、完成率重算 |
| 推荐算法模块 | server.js: calcRecommendedScore | 帖子推荐分实时计算 |
| 用户模块 | server.js: /api/users/* | 个人主页、排名查询 |
| 数据访问层 | src/db.js | 连接池管理、建表、事务封装 |

#### 2.3.2 子系统职责说明

- **认证模块**：调用微信 `jscode2session` 接口换取 openid；开发环境自动降级（以昵称为 key 构造 `dev_*` openid）。新用户返回 `isNewUser: true`，由前端引导绑定。
- **帖子管理模块**：维护帖子状态机（招募中 → 进行中 → 待评价 → 已完成 / 已放弃），每次读写帖子时调用 `syncPostStatus` 检查时间触发条件。
- **积分结算模块**：使用数据库事务保证"状态更新 + 积分变更"的原子性，防止网络闪断导致数据不一致。
- **推荐算法模块**：综合发布者信用（完成率×0.6 + 积分÷10×0.4）、帖子质量（奖励分 + 惩罚分 + 热度加成）和用户偏好（历史参与类别）三维加权，实时为每篇帖子计算推荐分。

### 2.5 关键技术与设计决策

#### 2.5.1 技术栈选择

| 层次 | 技术 | 版本 | 选择理由 |
|------|------|------|----------|
| 前端 | 微信小程序（WXML/WXSS/JS） | 基础库 3.x | 目标用户使用微信，零下载成本 |
| 后端框架 | Node.js + Express | Node 18+，Express 4.x | 异步 I/O 性能好，生态成熟 |
| 数据库 | MySQL 8.0 | 8.0+ | 支持事务、外键、复杂 JOIN，满足一致性要求 |
| ORM/驱动 | mysql2/promise | 3.x | 原生 Promise，支持连接池和预处理语句 |
| HTTP 客户端 | axios | 1.x | 调用微信开放平台接口 |

#### 2.5.2 重要设计决策记录

**决策 1：帖子状态由服务端主动同步（`syncPostStatus`）而非定时任务**

- 问题：帖子超时需自动流转状态（进行中 → 待评价），但小程序不能长驻后台运行定时器。
- 决策：每次读写帖子时在服务端调用 `syncPostStatus`，检查 `endTime` 和 `evaluationDeadline`，按需流转。
- 权衡：实现简单，无需额外进程；但状态变更不实时，需用户触发请求才生效。

**决策 2：积分结算使用数据库悲观锁事务（`withTransaction`）**

- 问题：积分是核心激励数据，要求"任务完成 → 积分增加"原子执行。
- 决策：通过 `UPDATE posts SET status='已完成' WHERE id=? AND status='待评价'` 的原子 UPDATE 实现乐观锁，`affectedRows === 0` 时跳过，防止重复结算。
- 权衡：避免分布式锁复杂度，适合单机部署场景。

**决策 3：推荐分在 API 请求时实时计算（内存排序），不持久化**

- 问题：推荐分依赖发布者当前积分/完成率和调用者的历史偏好，是动态值。
- 决策：取全量候选帖子到内存，实时计算后排序分页，不写回数据库 `recommendedScore` 字段。
- 权衡：帖子量小（校园场景）时性能可接受；量大时需改为异步预计算。

#### 2.5.3 技术风险分析

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 微信 `jscode2session` 超时或失败 | 用户无法登录 | 设置 3 秒超时，开发环境自动降级为 `dev_*` openid |
| 积分重复结算 | 数据不一致 | 乐观锁：原子 UPDATE + affectedRows 检查 |
| 内存推荐排序性能 | 帖子量大时响应慢 | 校园场景帖子量有限；长期方案：异步预计算 + 缓存 |
| MySQL 连接耗尽 | 服务不可用 | 连接池 `connectionLimit: 10`，队列等待模式 |

#### 3.1.1 目标用户特征

| 属性 | 描述 |
|------|------|
| 年龄 | 18–22 岁在校大学生 |
| 设备 | iOS 12+ / Android 8.0+，微信 7.0+ |
| 使用场景 | 自习组队、晨跑打卡、考研复习、求职备战 |

## 4 详细设计

### 4.1 用例详细设计

小组自选 3 个核心用例：**发布任务贴**（R03）、**任务完成判定与积分结算**（R10–R15）、**智能推荐排序**（R21）。

#### 4.1.1 用例 1：发布任务贴（R03）

**执行步骤**：

1. 用户在发布页填写表单（标题、内容、类别、积分、时间、最大搭子数）。
2. 前端校验：标题和内容不为空，积分 ≥ 0，结束时间 > 开始时间。
3. 前端调用 `api.createPost(payload)`，POST `/api/posts`。
4. 后端校验发布者存在（查 `users` 表）。
5. 后端生成帖子 ID（`p_{timestamp}_{random6}`），写入 `posts` 表，初始 `status = '招募中'`，`recommendedScore = 80`。
6. 返回完整帖子对象（含 `id`），前端刷新首页列表。

**前置条件**：用户已登录（`currentUserId` 非空）。

**后置条件**：`posts` 表新增一条记录，状态为「招募中」。

**异常处理**：

| 异常场景 | 处理方式 |
|----------|----------|
| 标题或内容为空 | 前端拦截，提示"标题和内容不能为空" |
| 发布者 ID 不存在 | 后端返回 400，前端 toast 提示 |
| 数据库写入失败 | 后端返回 500，前端提示"发布失败，请重试" |

#### 4.1.2 用例 2：任务完成判定与积分结算（R10–R15）

**执行步骤**：

1. 任务截止时间到达后，`syncPostStatus` 将状态从「进行中」变为「待评价」，并设置 `evaluationDeadline = now + 32h`。
2. 参与者（发布者和每位搭子）各自提交证据（POST `/evidence`），系统以 `(postId, submitterId)` 为唯一键，支持覆盖更新。
3. 参与者互相提交评价（POST `/evaluate`）：
   - 校验：`fromId ≠ toId`，双方均为参与者，评分 1–5，不可重复评价同一人。
   - 插入 `evaluations` 记录。
   - 调用 `updateUserAvgScore(toId)` 更新被评价者均分。
4. 判定全员评价完成：检查是否所有 `(fromId, toId)` 配对均有记录（两人情况即发布者→搭子、搭子→发布者各一条）。
5. 最后一条评价提交后，事务内原子执行：
   - `UPDATE posts SET status='已完成', progress=100 WHERE id=? AND status='待评价'`（乐观锁）
   - `UPDATE users SET points = points + reward WHERE id = publisherId`
   - 遍历 `post_buddies`，同样为每位搭子加积分
6. 调用 `recalcCompletionRate(publisherId)` 重算完成率。

**前置条件**：帖子状态为「待评价」，当前用户为参与者。

**后置条件**：帖子状态为「已完成」，发布者和搭子积分均增加，完成率重算。

**异常处理**：

| 异常场景 | 处理方式 |
|----------|----------|
| 重复评价同一人 | 后端返回 400，前端 toast 提示"已评价过" |
| 评分不在 1–5 | 后端返回 400"评分须为 1-5 的整数" |
| 乐观锁冲突（并发结算） | `affectedRows === 0`，跳过，防止重复加分 |
| 评价截止时间过期 | `canEvaluate = false`，前端隐藏评价按钮 |

#### 4.1.3 用例 3：智能推荐排序（R21）

**执行步骤**：

1. 前端调用 `api.getFeed({ userId, ... })`，GET `/api/posts?userId=xxx`。
2. 后端查询全量候选帖子（含发布者信息，LEFT JOIN users）。
3. 若 `userId` 存在，查询该用户历史参与帖子的类别分布（`doneCount`、`abandonCount`）构建 `preferenceMap`。
4. 对每篇帖子调用 `calcRecommendedScore(post, publisherUser, preferenceMap)` 计算推荐分（详见 4.4 算法设计）。
5. 按推荐分降序内存排序，截取当前页数据返回。
6. 冷启动（无历史数据）时，`prefScore` 默认 50，退化为基于帖子质量和发布者信誉的排序。

**前置条件**：无（游客可浏览，不传 userId 则无个性化偏好）。

**后置条件**：返回按推荐分排序的帖子分页列表，每条附带动态 `recommendedScore`。

**异常处理**：

| 异常场景 | 处理方式 |
|----------|----------|
| userId 不存在于数据库 | preferenceMap 为 null，退化为质量排序 |
| 数据库查询超时 | Express 错误中间件返回 500 |

### 4.2 类设计

#### 4.2.1 类识别与定义

系统共识别以下核心类（对应后端业务实体）：

| 类名 | 职责 |
|------|------|
| User | 用户信息、积分、完成率、AI 评价 |
| Post | 任务贴信息及状态机 |
| PostBuddy | 搭子关系记录 |
| Evidence | 任务完成证据 |
| Evaluation | 参与者互评记录 |
| ApiService（前端） | 封装所有 HTTP 请求，返回 Promise |
| PostController（后端） | 帖子相关路由处理，协调业务逻辑 |
| ScoreEngine（后端） | 推荐分计算、积分结算、完成率重算 |
| DbAdapter（后端） | 连接池管理、query/withTransaction 封装 |

#### 4.2.2 类关系说明

- `User` 1 ── * `Post`：一个用户可发布多篇帖子（publisherId 外键）。
- `Post` 1 ── * `PostBuddy`：一篇帖子可有多个搭子（postId 外键）。
- `User` 1 ── * `PostBuddy`：一个用户可加入多篇帖子（userId 外键）。
- `Post` 1 ── * `Evidence`：一篇帖子可有多条证据（每参与者一条，唯一约束）。
- `Post` 1 ── * `Evaluation`：一篇帖子可有多条互评（唯一约束 fromId+toId）。
- `PostController` 依赖 `ScoreEngine`（调用 settlePost、calcRecommendedScore）。
- `PostController` 依赖 `DbAdapter`（执行 SQL）。

**Post 类（对应 posts 表）**

| 属性 | 类型 | 说明 |

#### 4.3.3 数据存储与访问策略

- **连接池**：`connectionLimit: 10`，`queueLimit: 0`（无限排队等待），字符集 utf8mb4，支持中文及 emoji。
- **预处理语句**：所有 SQL 通过 `connection.execute(sql, params)` 执行，防止 SQL 注入。
- **外键级联**：`posts` 删除时级联删除 `post_buddies`、`evidences`、`evaluations`（ON DELETE CASCADE）。
- **索引策略**：
  - `users.openid` UNIQUE 索引（登录查询）
  - `post_buddies.(postId, userId)` UNIQUE（加入校验）
  - `evidences.(postId, submitterId)` UNIQUE（证据唯一性）
  - `evaluations.(postId, fromId, toId)` UNIQUE（防重复评价）
  - `evaluations.toId` 普通索引（收到的评价列表查询）
- **兼容性迁移**：`initDb` 启动时检查列是否存在（`SHOW COLUMNS FROM ... LIKE '...'`），缺失则 ALTER TABLE 补加，实现零停机迁移。

#### 4.4.3 关键算法：推荐分计算（`calcRecommendedScore`）

**输入**：帖子对象 `post`，发布者信息 `publisherUser`，用户偏好向量 `preferenceMap`

**输出**：0–100 的整数推荐分

**伪代码**：

```
function calcRecommendedScore(post, publisherUser, preferenceMap):
    // 发布者信用分（0–100）
    cr = publisherUser ? publisherUser.completionRate : 0       // 0–100
    pts = publisherUser ? min(publisherUser.points / 10, 100) : 0
    publisherScore = cr * 0.6 + pts * 0.4                       // 0–100

    // 帖子质量分
    rewardScore = min(post.reward / 2, 50)                      // 0–50
    penaltyScore = min(post.penalty / 2, 30)                    // 0–30
    hotBonus = (post.currentBuddies >= post.maxBuddies * 0.8) ? 20 : 0
    postScore = rewardScore + penaltyScore + hotBonus            // 0–100

    // 用户偏好分（0–100）
    pref = preferenceMap ? preferenceMap.get(post.category) : null
    if pref != null and (pref.doneCount * 2 + pref.abandonCount) > 0:
        prefScore = (pref.doneCount * 2) / (pref.doneCount * 2 + pref.abandonCount) * 100
    else:
        prefScore = 50   // 冷启动默认

    return round(publisherScore * 0.4 + postScore * 0.3 + prefScore * 0.3)
```

**设计说明**：
- 完成率权重（0.6）高于积分权重（0.4），因为完成率是更稳定的信用指标。
- 偏好分使用成功率公式（完成计 2 分、放弃计 1 分），给完成任务的用户更高权重。
- 热度加成（80% 满员时 +20 分）鼓励快速组队。
- 三维权重（0.4 + 0.3 + 0.3 = 1.0），产出 0–100 区间。

## 5 设计总结

### 5.1 设计成果概述

本次设计完整覆盖了 Task Buddy 三个设计阶段：

1. **系统架构设计**：确立微信小程序 + Express + MySQL 前后端分离三层架构，完成 6 个前端模块和 8 个后端模块的划分，定义全部 REST API 接口契约。
2. **界面设计**：制定校园清新风格规范，设计 6 个核心页面的原型布局，梳理 3 条核心交互流程（均在 3 次点击内完成）。
3. **详细设计**：细化 3 个核心用例的完整执行流程，完成 5 张数据库表的逻辑结构设计，设计推荐分算法和积分结算事务逻辑。

### 5.2 设计优势与不足

**优势**：

- 帖子状态机设计完整，5 种状态的转换条件和副作用均有明确约束。
- 积分结算采用乐观锁机制，在单机场景下有效防止并发重复结算。
- 推荐算法三维加权，兼顾冷启动（无偏好数据）和个性化两种场景。
- 数据库设计遵循范式，冗余字段（publisherName、buddyName）有明确的业务理由（减少 JOIN 次数）。

**不足**：

- 推荐排序采用全量内存排序，当帖子量大时性能下降，需改为数据库索引排序或预计算。
- 帖子状态同步依赖请求触发（`syncPostStatus`），状态变更不实时，高严格性场景需引入定时任务。
- 聊天功能（R09）在当前架构中仅有需求描述，未实现 WebSocket 或轮询方案，是后续版本的主要欠缺。
- 微信内容安全接口的集成在本文档中已规划，但在当前实现中尚未在帖子发布流程中调用。

### 5.3 后续工作展望

1. 接入微信内容安全 API，在 `POST /api/posts` 中增加文本审核步骤。
2. 实现搭子聊天功能（R09），推荐使用 WebSocket 或微信订阅消息方案。
3. 优化推荐算法性能：将 `recommendedScore` 异步预计算并存库，读时直接 `ORDER BY recommendedScore DESC`。
4. 引入定时任务（Node.js `node-cron`）定期调用 `syncPostStatus`，实现帖子状态自动流转。
5. 完善 AI 评价模块（R20）：基于积分变动、完成率、均分生成结构化文字评价。
